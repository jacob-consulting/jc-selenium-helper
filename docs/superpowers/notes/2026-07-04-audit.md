## Fixes Shipped (post-audit, 2026-07-04)

Work done after the audit below. The original findings are left intact as a record;
this log is the authoritative delta. **Released: v0.2.0 and v0.3.0 (both live on PyPI).**

1. ✅ **Insecure Chrome flags are now opt-in** (Security-Medium; Open Question #2, M1.2)
   — shipped in **v0.2.0**. `jc_chrome_options` no longer adds `--no-sandbox` /
   `--ignore-certificate-errors` by default; opt in via the new `jc_insecure_chrome`
   fixture or the `JC_SELENIUM_INSECURE` env var, which emits an
   `InsecureChromeOptionsWarning`. Documented in `pytest_plugin.md` + CHANGELOG.
   Open Question #2 resolved by the owner: *make it opt-in with a warning* (not
   document-only). This is a breaking change for CI adopters, hence the minor bump.

2. ✅ **Removed the non-deterministic `chrome_options` plugin fixture** — shipped in
   **v0.3.0**. A latent bug the audit did **not** catch: the plugin defined a
   `chrome_options` fixture to auto-feed `jc_chrome_options` into pytest-selenium, but
   pytest-selenium *also* defines `chrome_options`, and plugin-vs-plugin fixture
   precedence is not deterministic across environments. Result: the wiring worked
   locally but silently used pytest-selenium's plain defaults in CI — and the plugin
   tests were green locally / **red in CI on every recent master run** (including the
   pre-session commit `28ad61b`). Fix: drop the plugin fixture; users add a one-line
   `chrome_options` override to their own `conftest.py` (conftest fixtures reliably win).
   Tests rewritten to verify the pattern *with pytest-selenium installed*. See the new
   memory note `no-plugin-chrome-options-fixture`. **CI is now green** (lint / test 3.12
   + 3.13 / docs).

3. ✅ **GitHub Actions bumped off the deprecated Node 20 runtime** (DevEx; not in the
   original audit — surfaced from publish-run deprecation warnings). All actions moved to
   Node-24 majors: `checkout@v7`, `setup-python@v6`, `setup-chrome@v2`,
   `upload/download-artifact@v7`, and `setup-uv@v8.2.0`. **Gotcha recorded:**
   `astral-sh/setup-uv` publishes no moving `v8` major tag (only full `v8.x.y`), unlike
   the `actions/*` org — the first `publish.yml` release failed on `setup-uv@v8` and was
   fixed by pinning `@v8.2.0`. `publish.yml` can only be exercised by a real release, so
   this wasn't caught until the v0.3.0 publish.

4. ✅ **Hardened the version smoke test** — `tests/test_package.py` no longer hardcodes
   the version string; it reads `project.version` from `pyproject.toml`, so future
   `bump-my-version` runs can't break it.

**Still open** (unchanged by this session): M1.1 (`legacy.switch_and_fill_frame`),
QW1 (`original_source/` gitignore), QW2 / M2.2 (Python 3.14 in CI matrix — still
3.12/3.13 only), M0/M2 (coverage reporting + floor), and the M3 polish items.

## Executive Summary

**Overall health grade: A− (strong small library, a few honest gaps).**

This is a small (461 SLOC across 7 modules), well-scoped, well-executed Python
library that wraps Selenium with ergonomic helpers, plus optional pytest / Dynaconf /
axe extras. It is published to PyPI (v0.1.0), documented on Read the Docs, linted,
type-marked (`py.typed`), and CI-gated on lint + tests + docs. The code is clean,
consistent, and readable; the design (thin wrapper, optional extras isolated behind
extras, a `legacy` adapter for migration) is thoughtful and appropriate for a "Beta"
library. There are no critical or high-severity correctness or security defects.

The gaps are the expected ones for a fresh 0.1.0: **test coverage is execution-biased
and skips most failure paths and several public methods**; **the project claims Python
3.14 support it never exercises in CI**; **the shipped default Chrome options silently
disable the sandbox and TLS verification**; and **one legacy method (`switch_and_fill_frame`)
is broken by design** because it depends on an external object shape that no longer
exists. None of these block use; all are cheap to close.

- **Top 3 risks:** (1) `legacy.switch_and_fill_frame` cannot run — references
  `combined.settings.k_pause` on a caller object that isn't part of this package, and is
  untested; (2) default Chrome options ship `--no-sandbox` + `--ignore-certificate-errors`,
  a security posture consumers inherit silently; (3) Python 3.14 is advertised
  (classifiers + noxfile) but not in the CI matrix — a claim with no evidence.
- **Top 3 opportunities:** (1) add a coverage gate + fill failure-path and
  missing-method tests (biggest quality lever, mostly mechanical); (2) align the
  version-support story (classifiers ↔ CI ↔ noxfile) and add 3.14 to CI; (3) document
  / soften the insecure default Chrome flags.

## Repo Map

**Purpose:** A reusable Selenium test-helper library extracted and generalized from an
internal German-named test suite (see `original_source/`). Intended users are developers
writing browser tests. Maturity: freshly published library, `Development Status :: 4 - Beta`.

**Stack:** Pure Python (`py3-none-any`), `requires-python >=3.12`. Runtime dep:
`selenium>=4.38`. Optional extras: `pytest`+`pytest-selenium`, `dynaconf`, `axe-selenium-python`.
Build: setuptools + `src/` layout. Tooling: ruff (lint+format), pre-commit, nox,
bump-my-version, mkdocs (Read the Docs). CI: GitHub Actions (lint/test/docs) + PyPI
Trusted Publishing on release.

**Architecture sketch:** One primary class, `Browser` (`browser.py`), holding a Selenium
`driver` and calling through to it with convenience methods (finders → waits → actions →
assertions → frames). Everything else is a thin, independent leaf module:
`colors` (pure function), `config` (Dynaconf loader), `axe` (a11y), `plugin` (pytest
fixtures), `legacy` (deprecating adapter subclass of `Browser`). No cross-coupling between
leaf modules; no circular deps. Control flow is entirely synchronous/blocking (Selenium's
sync API). No god objects, no layering to violate.

**Key directories:**
| Path | Description |
|---|---|
| `src/jc_selenium_helper/` | The library (7 modules, 461 SLOC). |
| `tests/` | 11 test modules, live-Chrome + `pytester` + pure-unit; HTML fixtures in `tests/fixtures/`. |
| `docs/` | mkdocs site: getting-started + per-module reference + migration guide. |
| `docs/superpowers/` | SDD spec, plan, this audit (excluded from the published docs site). |
| `.github/workflows/` | `ci.yml` (lint/test/docs), `publish.yml` (OIDC PyPI publish). |
| `original_source/` | **Untracked.** The pre-refactor internal suite this package generalizes. |

**What surprised me (facts):**
- `original_source/` (the internal source material, incl. `settings.global.yaml`,
  `settings_testuser.yaml`) is **untracked and not in `.gitignore`** — one `git add -A`
  from being committed to a public repo. It contains no credentials (test persona data
  only), but it's an org-internal artifact that shouldn't live in a public checkout.
- The `legacy.switch_and_fill_frame` method (`legacy.py:146`) takes a `combined` argument
  and reads `combined.settings.k_pause` — an object shape from the *original* suite, not
  this package. It is preserved "verbatim" but is effectively dead/unrunnable here.
- 3.14 appears in classifiers (`pyproject.toml:15`) and `noxfile.py:4` but not in the CI
  matrix (`ci.yml:24` → only 3.12/3.13).

## Audit Report

Findings are grouped by dimension, sorted by severity within each. Each is labelled
**[Fact]** (mechanically verifiable) or **[Judgment]** (defensible opinion).

### Architecture & design — *mostly healthy*

- **[Fact] Low — Inconsistent wait-vs-immediate semantics across assertions.**
  `assert_checkbox_checked`/`assert_checkbox_unchecked` wait (`wait_present`,
  `browser.py:152,156`), but `assert_selected_option` (`browser.py:159–163`) and
  `assert_present` (`browser.py:165–167`) resolve immediately via `find`/`exists` with no
  wait and no `timeout` param. Consequence: an element that appears slightly late makes
  `assert_selected_option`/`assert_present` flake while the checkbox asserts would have
  waited. The asymmetry is undocumented.
- **[Judgment] Low — `Browser` is a mild "grab bag."** 21 public methods spanning five
  concerns on one class. This is acceptable and idiomatic for a page-object base at this
  size, but it is the class most likely to accrete. No action needed now; worth watching.

### Code quality — *good, small nits*

- **[Fact] Medium — `legacy.switch_and_fill_frame` is unrunnable and untested**
  (`legacy.py:146–154`). It calls `time.sleep(combined.settings.k_pause)` on a `combined`
  parameter that has no source inside this package, and hardcodes `//*[@id='tinymce']/p`.
  Any caller of this "compatibility" method gets an `AttributeError`. It exists only to
  preserve a signature but cannot deliver the compatibility it advertises.
- **[Fact] Low — `hover_with_offset` performs two separate action chains**
  (`browser.py:112–116`): `move_to_element(...).perform()` then `move_by_offset(...).perform()`.
  Because the second `perform` starts from the pointer's current (post-move) position in a
  fresh sub-action, the resulting offset behavior differs from a single chained
  `move_to_element(el).move_by_offset(x,y).perform()`. Likely not what a reader expects;
  untested, so the intended behavior is unverifiable.
- **[Fact] Low — `rgb_to_hex` has no input validation** (`colors.py:11–14`). Malformed
  input raises a bare `ValueError`/`IndexError` from `str.index`/`int(float(...))` with no
  helpful message. Minor; it's a test helper.
- **[Fact] Low — `wait_document_ready` busy-polls with `time.sleep`** (`browser.py:68–75`)
  rather than `WebDriverWait`, and the `elapsed > seconds` check can overshoot the timeout
  by up to one `poll_pause`. Functionally fine; slightly inconsistent with the
  `WebDriverWait`-based sibling waits.

### Security — *acceptable for a test helper, one thing to surface*

- **[Fact] Medium — Default Chrome options disable the sandbox and TLS verification.**
  ✅ **Resolved 2026-07-04.** `plugin.py` originally shipped `--no-sandbox` and
  `--ignore-certificate-errors` as *default* `jc_chrome_options`, so any project adopting
  the fixture silently ran with the sandbox off and certificate errors ignored. These are
  now **opt-in**: the two insecure flags are added only when the `jc_insecure_chrome`
  fixture returns `True` or the `JC_SELENIUM_INSECURE` env var is set, and opting in emits
  an `InsecureChromeOptionsWarning`. Safe-by-default; documented in `pytest_plugin.md` and
  the CHANGELOG. (Same flags remain in `tests/conftest.py` — fine there, it's this repo's
  own CI.)
- **[Fact] Low — `original_source/` is untracked but not git-ignored.** Contains internal
  org settings/persona files. No secrets were found (grep for pass/secret/token/key across
  the YAML returned only comments and test data), but it risks accidental commit to a
  public repo. Add to `.gitignore` or remove.
- **[Fact — healthy] No hardcoded credentials in tracked code.** `publish.yml` uses OIDC
  Trusted Publishing (no stored token). No `eval`/`pickle`/`os.system`/`subprocess` shell
  injection surfaces in the library.

### Testing — *the main weak spot*

- **[Fact] High — Failure paths and several public methods are untested.** 37 tests exist
  and assert real behavior (not just execution) where present, but coverage is happy-path
  biased. Untested public API: `wait_not_present`, `wait_move_click`, `hover_with_offset`,
  `upload_file`, `move_to` (asserts nothing meaningful — `test_move_to_does_not_raise`
  only checks no-raise), `wait_page_loaded` failure/retry path, `fill_in_frame` with
  `submit=True`, `assert_checkbox_unchecked` fail path, and the `jc_browser` fixture
  (`test_plugin.py` covers only `chrome_options`). Legacy `seite_geladen` and
  `switch_and_fill_frame` are untested (the latter is unrunnable — see above).
  *(Update: the thin plugin-fixture coverage referenced here masked a real
  non-deterministic bug that only surfaced in CI — found and fixed in v0.3.0; see
  Fixes Shipped #2. `jc_browser` and the failure-path gaps remain open.)*
- **[Fact] Medium — No coverage measurement or gate.** `pytest-cov` is in the `test`
  extra (`pyproject.toml:43`) but `addopts` (`pyproject.toml:68`) never invokes it and CI
  (`ci.yml:32`) runs bare `pytest -q`. There is no floor, so coverage can silently regress.
- **[Fact] Low — Two "does not raise" tests assert nothing about outcome**
  (`test_browser_actions.py:20–27`). `hover`/`move_to` tests only confirm no exception,
  not that the pointer moved. Low value; acceptable given Selenium's opacity here.
- **[Judgment] Low — Live-Chrome tests are inherently slower/flakier** than the
  `pytester`/pure-unit tests, but the session-scoped `driver` fixture and local
  `file://` fixtures keep this well-managed. No action needed.

### Performance — *healthy*

- One sentence: no performance concerns. This is a thin synchronous wrapper over
  Selenium's own network-bound I/O; there are no queries, no async paths, no caches, and
  no unbounded growth. `time.sleep` usage is intrinsic to Selenium interaction pacing.

### Dependencies — *healthy*

- **[Fact — healthy] Lean and current.** One runtime dep (`selenium>=4.38`); all others
  are behind extras. No lockfile is committed (appropriate for a library — callers
  resolve). `pre-commit` pins `ruff-pre-commit` to `v0.8.0` (`.pre-commit-config.yaml:3`)
  while `dev`/`test` extras use unpinned `ruff` — a minor drift risk (**Low**), not a
  problem.

### DevEx & operations — *strong*

- **[Fact] Medium — CI matrix contradicts the advertised Python support.** Classifiers
  (`pyproject.toml:13–15`) and `noxfile.py:4` claim 3.12/3.13/3.14; the CI test matrix
  (`ci.yml:24`) runs only 3.12/3.13. 3.14 is asserted to users with zero CI evidence.
  *(Still open.)*
- **[Fact] Low — GitHub Actions were on the deprecated Node 20 runtime.**
  ✅ **Resolved 2026-07-04** (not in the original audit; surfaced from publish-run
  deprecation warnings). All actions bumped to Node-24 majors — see Fixes Shipped #3.
- **[Judgment — healthy] Otherwise excellent for the size:** lint+format enforced in CI
  and pre-commit, docs built `--strict`, OIDC publish, bump-my-version wired, nox for
  local multi-version runs. Setup friction is low (`pip install -e ".[test]"`).
- **[Fact] Low — No observability, by design.** The library has no logging. Fine for a
  test helper; noted only for completeness.

### Documentation — *strong, verified accurate*

- **[Fact — healthy] README, CHANGELOG, and reference docs match the code.** Spot-checked
  `docs/reference/config.md` against `config.py` (signature, prefixes, env switcher all
  correct) and the CHANGELOG method list against `browser.py` (complete and accurate).
  `superpowers/` is correctly excluded from the published site (`mkdocs.yml`).
- **[Fact] Low — The insecure default flags are undocumented.** `docs/reference/pytest_plugin.md`
  (and README) do not warn that the default `jc_chrome_options` disable the sandbox and TLS
  verification. Pairs with the Security-Medium finding.

### Strengths (preserve these)

1. **Clean, consistent, readable code** — uniform method shape, sensible XPath-default
   convention, docstrings that explain *why* (e.g. the `jc_` fixture-prefix rationale).
2. **Well-isolated optional extras** — `axe`/`config`/`pytest` imports are confined to
   their own modules; `__init__` imports only `browser`, so core install stays minimal.
3. **Thoughtful migration story** — the `legacy` adapter + migration guide let existing
   suites move with a one-line import swap and get `DeprecationWarning`s pointing at the
   new names.
4. **Mature release infrastructure for a 0.1.0** — OIDC publishing, strict docs, lint
   gates, multi-version nox, semantic version bumping.

## Improvement Strategy

**Themes (each explains a cluster of findings):**

1. **Tests prove things work, not that they fail correctly, and don't cover the whole
   surface.** *Target state:* every public method has at least one happy-path and one
   failure/edge test; a coverage floor prevents regression. *Principle:* a test suite's
   value is in the branches it exercises, not the lines it touches.
2. **The support/compatibility story makes claims it doesn't back with evidence.** Python
   3.14 (claimed, not CI-tested) and `legacy.switch_and_fill_frame` (advertised as
   compatible, actually unrunnable). *Target state:* every advertised capability is either
   exercised in CI or explicitly marked provisional/removed. *Principle:* don't promise
   what you don't verify.
3. **Convenient-but-insecure defaults are shipped silently.** *Target state:* the sandbox
   / TLS-bypass flags are either opt-in or loudly documented at the point of use.
   *Principle:* security-relevant defaults must be visible to the person inheriting them.
4. **Minor internal inconsistencies** (wait-vs-immediate asserts, double-perform offset,
   busy-poll ready check). *Target state:* uniform semantics, documented where they differ.
   *Principle:* least surprise.

**Explicit non-goals (recommend NOT doing now, with reasons):**
- **Don't split `Browser` into page-object mixins.** At 21 methods / one file it's within
  reason; splitting adds import ceremony for no user benefit at this maturity.
- **Don't add logging/observability.** It's a test helper invoked from a test runner that
  already reports; adding a logging layer is pure overhead.
- **Don't pin a lockfile.** Libraries should float their deps; a lockfile would mislead
  consumers.
- **Don't rewrite `wait_document_ready` to `WebDriverWait`** unless touching that area
  anyway — it works; the payoff is cosmetic.

**Definition of done (measurable signals):**
- CI runs `pytest --cov` with `--cov-fail-under=85` (or an agreed floor) and passes.
- CI test matrix includes every Python version in the classifiers (3.12/3.13/3.14) **or**
  the 3.14 classifier is removed.
- Zero unrunnable public/legacy methods (fix or remove `switch_and_fill_frame`).
- The insecure default Chrome flags are documented at every entry point that ships them.
- `original_source/` is git-ignored or removed.

### Implementation sketches — top 3 tasks

**M0.2 — Backfill failure-path & missing-method tests (biggest quality lever).**
*Approach:* mirror the existing `browser` + `fixture_url` fixture style; add fixtures/HTML
where an element appears late or disappears (small inline `<script>setTimeout(...)>`).
*Key steps:* (1) add a `late.html` with a delayed-append element to drive `wait_not_present`
and the `assert_*` late-appearance cases; (2) test `wait_page_loaded` retry path against a
never-appearing selector with `retries=1, interval=0.1` and assert `TimeoutError`; (3) test
`upload_file` by pointing a `file://` `<input type=file>` at a fixture file and asserting
the `value`; (4) test `jc_browser` via `pytester` like the existing `chrome_options` tests.
*Gotchas:* keep timeouts tiny (existing `browser` fixture uses `default_timeout=10`,
`poll_pause=0.2`); `upload_file` hardcodes `By.CSS_SELECTOR` — pass a CSS selector, not XPath.

**M1.1 — Fix or remove `legacy.switch_and_fill_frame`.**
*Approach:* it cannot delegate cleanly because `combined.settings.k_pause` has no analogue
here. Preferred: **delete it** and add a migration-guide note (callers should move to
`fill_in_frame`, passing their own pause explicitly); alternatively keep the name but change
the signature to `(frame_path, inner_path, text, pause=0)` and delegate to `fill_in_frame`.
*Key steps:* pick the path with the owner (see Open Questions), implement, add one test,
update `docs/reference/migration.md`. *Gotchas:* this is the only legacy method whose
signature genuinely can't be preserved, so "verbatim compatibility" is already broken —
removing it is honest, not a regression.

**M1.2 — Sandbox/TLS-bypass default flags.**
*Approach:* the cleanest is to keep the CI-friendly defaults (adopters overwhelmingly run
in CI) but make the tradeoff explicit: expand the `jc_chrome_options` docstring and
`pytest_plugin.md` to state that `--no-sandbox` and `--ignore-certificate-errors` are on by
default and how to override (redefine `jc_chrome_options` in the project conftest — already
supported and tested). *Key steps:* edit docstring + docs; optionally add a
`jc_chrome_options`-without-insecure-flags note. *Gotchas:* actually *removing* the flags
would break many consumers' CI (headless Docker needs `--no-sandbox`), so prefer
documentation over silent behavior change — hence coupling with QW3.
