# M0.2 — `browser.py` test backfill (design)

**Date:** 2026-07-04
**Source:** audit task **M0.2** (`docs/superpowers/notes/2026-07-04-audit.md`)
**Type:** additive tests + fixtures only — **no production code changes**

## Overview

The audit rated test coverage the library's main weak spot: the suite is
happy-path biased and skips most failure paths and several public methods.
Coverage reporting (added in the second quick-wins pass) put `browser.py` at 46%.
This backfill adds behavior-asserting tests for the untested `browser.py` methods
and failure paths, forming the safety net before the M3.x correctness fixes.

Scope is deliberately narrowed to **`browser.py` only** (audit Option A). The
`legacy` adapter (M-follow-up) and the `jc_browser` fixture are out of scope here.

## Scope

**In scope** — new/upgraded tests for these `browser.py` methods:

- `wait_not_present` — success and timeout paths
- `wait_page_loaded` — retry-then-timeout path (happy path already tested)
- `wait_move_click` — click effect
- `upload_file` — file input value set
- `assert_checkbox_unchecked` — fail path (pass path already tested)
- `fill_in_frame(submit=True)` — RETURN is sent (only `submit=False` tested today)
- `move_to` — upgraded from no-raise to a real `mouseover` assertion
- `hover` — upgraded from no-raise to a real `mouseover` assertion

**Out of scope, with reasons:**

- `hover_with_offset` — deferred to audit task **M3.2**, which *fixes* it. Writing a
  characterization test now would pin behavior the audit already suspects is wrong,
  then require rewriting the test when M3.2 lands. Its test is authored together with
  the fix (proper TDD).
- `wait_document_ready` timeout branch — forcing `document.readyState` to never reach
  `"complete"` is not achievable over `file://` without a stalling HTTP server. Not
  worth a mock for one line; the happy path stays covered.
- `legacy.py` and `plugin.py` (`jc_browser`) coverage — separate follow-up.

## Test inventory

| # | Method | Test intent | Assertion | Fixture |
|---|--------|-------------|-----------|---------|
| 1 | `wait_not_present` (success) | element disappears | returns `True` | `dynamic.html` — `#vanishing` hidden ~300ms via JS |
| 2 | `wait_not_present` (timeout) | element stays present | raises `TimeoutException`, message contains "still present" | `basic.html` `#title`, `timeout=1` |
| 3 | `wait_page_loaded` (retry timeout) | selector never appears | raises `TimeoutError`, message contains "Page not loaded" | `basic.html`, `retries=1, interval=0.1` |
| 4 | `wait_move_click` | move-to then click | `#clickable` text becomes "clicked" | `basic.html` |
| 5 | `upload_file` | set file input | input `value` ends with the sent filename | `dynamic.html` `#upload` |
| 6 | `assert_checkbox_unchecked` (fail) | checkbox is checked | raises `AssertionError` | `basic.html` `#checked-box` |
| 7 | `fill_in_frame(submit=True)` | RETURN sent after text | inner `#submitted` marker equals the typed text | `frame.html` → enhanced `inner.html` |
| 8 | `move_to` (upgraded) | pointer moves onto element | `#hover-result` text becomes "hovered" | `dynamic.html` |
| 9 | `hover` (upgraded) | pointer moves onto element | `#hover-result` text becomes "hovered" | `dynamic.html` |

Tests #8 and #9 replace the existing weak `test_move_to_does_not_raise` and
`test_hover_does_not_raise` (which the audit flagged as asserting nothing).

## Fixtures

**New `tests/fixtures/dynamic.html`:**

- `#vanishing` — a visible element that JS sets to `display:none` after ~300ms
  (drives `wait_not_present` success; `invisibility_of_element_located` resolves on
  a hidden element).
- `#hover-target` + `#hover-result` — `#hover-target` has an `onmouseover` handler
  that writes `"hovered"` into `#hover-result` (drives `move_to`/`hover`).
- `#upload` — `<input type="file">` (drives `upload_file`, which uses
  `By.CSS_SELECTOR`).

**Extend `tests/fixtures/inner.html`:**

- Add a `keydown` listener on `#inner-input` that, on `Enter`, writes the field's
  current value into a new `#submitted` element. This is the observable signal that
  `fill_in_frame(submit=True)` actually sent RETURN (distinguishing it from
  `submit=False`).

**Untouched:** `basic.html`, `frame.html` (still references the enhanced
`inner.html`), and all existing fixtures used by current tests.

## Approach

- New tests slot into the existing per-concern files: `test_browser_waits.py`,
  `test_browser_actions.py`, `test_browser_assertions.py`, `test_browser_frame.py`.
- Reuse the existing `browser` and `fixture_url` fixtures from `tests/conftest.py`
  (`default_timeout=10`, `poll_pause=0.2`). No `conftest.py` changes.
- `upload_file` needs a real on-disk path: point it at an existing fixture file
  (e.g. `FIXTURES / "basic.html"`) and assert the input `value` ends with that
  filename.
- Keep timeouts/intervals tiny so the suite stays ~5-6s total.

## Acceptance criteria

- All 9 tests pass locally and in CI on Python 3.12/3.13/3.14 (live Chrome).
- The two weak no-raise tests are removed, replaced by the meaningful #8/#9.
- No changes under `src/`.
- `ruff check` + `ruff format --check` clean; `mkdocs build --strict` unaffected.
- `browser.py` coverage rises from 46% toward ~85-90% (remaining gap =
  `hover_with_offset` (M3.2) + `wait_document_ready` timeout branch).

## Non-goals

- No coverage floor / `--cov-fail-under` (that is audit task **M2.1**, after
  `legacy`/`plugin` are also covered).
- No production code changes; no refactors of `browser.py`.
